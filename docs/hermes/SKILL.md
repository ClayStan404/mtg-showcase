# MTG Showcase Query

> Restorable copy - the live file is at
> `~/.hermes/skills/domain/mtg-showcase/SKILL.md` on radxa-32g. If you change
> one, update the other.

Search the claystan.cc MTG marketplace for in-stock cards (sellers) and wanted
cards (buyers). The site has NO prices - it connects buyers and sellers; contact
info is included so they can deal off-site.

## When to Use

- User asks if a card is in stock / who sells X
- User asks who wants to buy X
- User asks what a seller has / cards in a city

## Data Sources (fetch fresh each time)

- In-stock (sellers): https://claystan.cc/data/cards.json
- Wanted (buyers):   https://claystan.cc/data/wants.json

Both are updated hourly by the site's auto-update pipeline. Fetch via web_request
or curl; do not cache across days.

## Schema

cards.json top-level: generated_at, site, count, total_quantity, sellers, cities, cards.
Each card in cards[]:
- name_en, name_zh, name_printed - card name (English / Chinese / printed)
- set, set_name, number - set code (e.g. "2x2"), full set name, collector number
- lang - en/zhs/ja/other; lang_label - display label (英文/中文/日文/其他)
- foil - boolean
- quantity - int
- seller, seller_id, city, contact - seller info
- image, scryfall_uri - card image + Scryfall link
- NO price field.

wants.json is symmetric: wants[] with buyer instead of seller, plus must
(bool: true = exact printing required). May be empty (count 0).

## How to Search

Match case-insensitively, support partial matches, on name_en / name_zh /
name_printed. Also filter by set, seller/buyer, city, foil, lang.

## Answer Format

List each match as:
<name_zh> (<name_en>) | <set_name> | <lang_label> <foil?> | x<quantity> | 卖家: <seller> (<city>) | 联系: <contact>

For wants, replace seller with 求购: <buyer> (<city>) | 联系: <contact>; note must=true if set.

Always state: 站点无价格信息，请直接联系卖家/买家询价。
If no match: 没找到 <X> 的在售/求购记录。

## Examples

User: "有没有 Sol Ring" -> fetch cards.json, filter by name, list matches.
User: "谁求购 Sol Ring" -> fetch wants.json, filter by name.
User: "ClayStan 都在卖什么" -> filter seller == "ClayStan".

## Operational notes for this machine

- **Direct HTTPS to claystan.cc fails** from a bare `curl` with `OpenSSL SSL_read: ... unexpected eof`. This host is behind an egress proxy.
- **Always fetch via the corporate proxy**: `curl -sS --max-time 20 -x http://shproxy.cixtech.com:3128 https://claystan.cc/data/cards.json -o /tmp/cards.json`
- If the corporate proxy is also unreachable, fall back to the local SOCKS/HTTP proxy: `-x http://127.0.0.1:1080`.
- Do not cache the JSON across days - `generated_at` shows real freshness, and the "hourly" promise often lags many hours when the upstream GitHub Actions workflow has a bad run.
- Stale `generated_at` + no recent GitHub Actions run = the companion watchdog cron is doing its job; check the upstream `auto-update.yml` workflow on `ClayStan404/mtg-showcase` rather than re-running things locally.

## Site pipeline context (for "why is the data stale?" questions)

`claystan.cc` data is generated by GitHub Actions on the `ClayStan404/mtg-showcase` repo:

1. `auto-update.yml` runs hourly, scrapes the source, commits `data/cards.json` + `data/wants.json`.
2. The companion cron job `mtg-auto-update-check` (script: `~/.hermes/scripts/mtg_auto_update_check.py`, no_agent, every 20 min) polls the Actions API and alerts when the most recent successful run is older than 2 h, or when the latest run failed/cancelled. It authenticates with a GitHub PAT from `MTG_GH_TOKEN` env var or `~/.config/gh/hosts.yml` (anonymous calls are rate-limited to 60/h and will appear as `urllib.error.HTTPError: HTTP Error 403: rate limit exceeded`).
3. If you see fresh `generated_at` but no recent Actions runs, the cron watchdog is wrong. If you see no recent runs and stale data, both are broken upstream.
